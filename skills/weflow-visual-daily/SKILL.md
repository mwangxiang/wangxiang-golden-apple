# WeFlow Visual Daily

## Layer

Candidate backstage skill. It may produce finished files, but prompt styles and visual templates remain candidates until repeated user approval.

## Contract

The deliverable is finished shareable PNGs, not only prompts.

When the request asks for both avatar/SBTI output and daily-report output, it means two separate PNG deliverables:

- `contentDaily`: an independent community content daily poster, generated locally from the selected messages.
- `imageModelFinal`: an independent SBTI/avatar portrait poster, generated by the image model and finalized from `.codex/generated_images`.

Never put the daily report inside the avatar/SBTI poster. A single combined PNG does not satisfy an avatar-plus-daily request.

- Prefer the image generation model for illustrated posters when image references can be provided.
- A final answer may present Downloads/project PNG paths only after `node scripts/validate-run.mjs --run-dir "reports/visual-daily/RUN" --require-download true` returns `ok: true`.
- If validation returns `ok: false`, the only acceptable final status is "not completed / failed validation" with the failed checks. Do not call the PNG final.
- For SBTI/avatar portrait posters, local vector, HTML, PowerShell, canvas, screenshot, or deterministic renderer outputs are proof/fallback artifacts only. They are not final unless the manifest has valid `imageModelFinal` evidence from `.codex/generated_images`.
- Every final PNG must be copied to both:
  - `reports/visual-daily/RUN/generated/`
  - `downloads/weflow-visual-daily-RUN/` under the local WeFlow tools root.
- Never call random generated faces an avatar-linked poster.

Unique entrypoint for SBTI/avatar portrait work:

```powershell
node scripts\sbti-avatar-pipeline.mjs prepare --run-dir "reports\visual-daily\RUN" --group "Group Name" --date "YYYY-MM-DD" --interval "HH:mm-HH:mm" --linkage-mode reference-image-conditioned
node scripts\sbti-avatar-pipeline.mjs finalize --run-dir "reports\visual-daily\RUN" --source-generated-image "C:\Users\YOU\.codex\generated_images\...\image.png" --name "final-sbti-avatar.png" --download-name "shareable-file-name.png"
node scripts\sbti-avatar-pipeline.mjs validate --run-dir "reports\visual-daily\RUN"
```

Use this entrypoint instead of calling local poster renderers directly. `prepare` never creates a final avatar poster; it produces the canonical image-model prompt, `READY_FOR_IMAGE_GEN.md`, and the separate content daily PNG when message/member sources are available. Use `reference-image-conditioned` when the avatar sheet can be uploaded to the image model; use `avatar-trait-linked` only when concrete per-avatar traits are available and image upload is unavailable.

## Workflow

1. Resolve group, absolute Asia/Shanghai date, and run slug.
2. Work from the local WeFlow tools root. Prefer a short ASCII path such as `D:\weflow-tools` for shareable setups.
3. Confirm WeFlow API health. If `5031` is not listening, clear `ELECTRON_RUN_AS_NODE` before starting `WeFlow.exe`.
4. Export JSON only through Node scripts:
   - `scripts/weflow-json-to-file.mjs`
   - never PowerShell `>`, `Out-File`, or `Tee-Object` for JSON.
5. Generate daily data, Top10/persona data, avatar references, prompt pack, and manifest.
6. Inspect `avatar-reference/top10-avatar-reference-sheet.png` before final avatar delivery.
7. For SBTI/avatar portrait posters, run `scripts\sbti-avatar-pipeline.mjs prepare`. This stage also creates the separate content daily PNG when sources exist.
8. Call the image model using the canonical prompt. If no image-model PNG is produced, stop at `READY_FOR_IMAGE_GEN.md` and report not completed.
9. Run `scripts\sbti-avatar-pipeline.mjs finalize` with the `.codex/generated_images` source.
10. Run `scripts\sbti-avatar-pipeline.mjs validate`. Only a passing result can be final; for avatar-plus-daily requests it must show two different files.

## Avatar Linkage

Avatar posters must be one of:

- `reference-image-conditioned`: the image model received the avatar sheet or per-person reference images.
- `avatar-trait-linked`: the prompt contains concrete per-avatar traits: subject/object, colors, pose, background, symbols, and emotion.
- `deterministic-avatar-linked`: the local renderer directly uses the real avatar files. This is fallback/proof only for improved-avatar requests.

Reject:

- plain random portraits
- a prompt that only says "use reference N" without attached images or trait descriptions
- WeChat placeholder avatars treated as real avatars
- content daily and avatar poster pointing to the same PNG

## GPT Image 2 SBTI Template Gate

When the user asks for a Top10/SBTI/avatar portrait poster, the final image-model prompt must be based on `references/gpt-image-2-sbti-template.md`.

Before calling the image model, create and validate:

```text
reports/visual-daily/RUN/image-model-pack/gpt-image-2-sbti-template-filled.md
```

The filled template must contain, for every real member card:

- `姓名`
- `发言数` and percent
- `SBTI类型`
- `插画场景` based on avatar reference traits plus behavior/persona
- `性格标签`
- `金句`

## Stability Rules

- In test/fix mode, repair scripts and skill rules first, then rerun before claiming success.
- In urgent delivery mode, produce the fastest usable PNG first, then record and fix workflow defects.
- `members_counts.json.messageCount` may be all-time. For a daily poster, Top10 must be counted from that day's `messages.json`.
- WeFlow desktop config is sensitive. Do not expose tokens or decrypt keys.
- If `httpApiToken` is stored as `safe:...`, do not edit the big config with PowerShell regex. If token repair is unavoidable, use Node `JSON.parse`/`JSON.stringify`, make a backup, and validate JSON immediately.
- Chinese `.ps1` scripts should be UTF-8 with BOM for Windows PowerShell 5.
- Validate UTF-8 files with Node and reject files containing `\uFFFD`.

## Output Shape

Return:

- Downloads PNG paths for each deliverable
- project `generated/` PNG paths for each deliverable
- for avatar-plus-daily requests, explicitly label the two separate deliverables as `contentDaily` and `avatarPoster`
- source group and absolute date
- validation result
- output mode: `reference-image-conditioned/final`, `avatar-trait-linked/final`, `deterministic-avatar-linked/fallback`, or rejected
- unresolved visual issues requiring approval
